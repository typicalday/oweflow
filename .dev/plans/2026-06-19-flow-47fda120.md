# PR2b — named-handler onInvalidate routing

## Goal

Make `effect: { idempotent: false, onInvalidate: '<loopName>' }` a valid loop declaration. When invalidation fires (L's green artifact's consumed input moves to a new version, L is non-idempotent, onInvalidate is a loop name H):
1. **Pin L's artifact** — keep it green, re-point fingerprint to current input versions (exactly as `onInvalidate: 'pin'` already does), so the cascade is stable.
2. **Arm H** — materialize H's produced output(s) as `owed` if absent, or re-arm to `owed` if present-and-green, so `eligibleFirings` picks H up next tick.

H is a normal forward producer loop — no Saga sequence, no new acceptance state. The engine detects + routes only.

## Detailed changes

### 1. `src/types.ts` — widen `EffectDef.onInvalidate`

**Current** (line ~170):
```ts
onInvalidate?: 'pin' | 'escalate';
```

**Change to:**
```ts
/** Routing when idempotent:false and an input moves.
 *  'pin'      — keep the artifact green, re-point fingerprint to current inputs.
 *  'escalate' — reject the artifact as held; producer not auto-re-eligible.
 *  '<loopName>' — pin the original AND arm the named handler loop (D-A/D-B).
 *  Defaults to 'escalate' when idempotent:false and omitted. */
onInvalidate?: 'pin' | 'escalate' | string;
```

Note: TypeScript's `string` supersedes the literals; the union is kept for documentation clarity. The practical guard is in defs.ts validation, not the type.

### 2. `src/defs.ts` — `buildLoop` (current lines ~314–324)

**Current block** that throws immediately for non-pin/escalate:
```ts
if (oi !== 'pin' && oi !== 'escalate') {
  throw new DefError(`loop '${name}': effect.onInvalidate must be 'pin' or 'escalate'; ...`);
}
effectDef.onInvalidate = oi;
```

**Replace with:** Store any string verbatim without throwing. Validation moves to `validateDef` where the full def is available for cross-reference checks.
```ts

```

**Current `validateDef` effect block** (lines ~473–483) does a belt-and-suspenders guard that repeats the `!= 'pin' && != 'escalate'` check. Replace it entirely with D-D cross-reference validation:

```ts
// effect: validation (D-D)
for (const l of def.loops) {
  if (!l.effect) continue;
  if (l.terminal && l.effect) {
    errors.push(`loop '${l.name}': terminal: true and effect: are mutually exclusive; effect: is the forward spelling — remove terminal: true`);
  }
  const oi = l.effect.onInvalidate;
  if (oi !== undefined && oi !== 'pin' && oi !== 'escalate') {
    // Named handler: D-D cross-reference checks
    const handlerLoop = def.loops.find((h) => h.name === oi);
    if (!handlerLoop) {
      errors.push(`loop '${l.name}': effect.onInvalidate '${oi}' names a loop that does not exist in this workflow`);
    } else if (oi === l.name) {
      errors.push(`loop '${l.name}': effect.onInvalidate '${oi}' names itself; a loop cannot be its own handler`);
    } else if (singletonProduces(handlerLoop).length === 0 && collectionProduces(handlerLoop).length === 0) {
      errors.push(`loop '${l.name}': effect.onInvalidate handler '${oi}' produces no outputs; a handler must produce at least one output`);
    }
  }
}
```

Import `singletonProduces` and `collectionProduces` from `./model.ts` at the top of defs.ts (or inline equivalent filter on `loop.produces`). Since defs.ts already imports from types.ts but not model.ts, prefer inlining: `handlerLoop.produces.filter(p => p.kind === 'singleton' || p.kind === 'collection')`. Note: defs.ts already imports `EffectDef` from types.ts; the import line does NOT need to import from model.ts.

Alternative (no new import): use `handlerLoop.produces.length === 0` — simpler and sufficient, since `map` produce patterns also count as outputs (handler must produce at least one artifact path to be armable). Use `handlerLoop.produces.length === 0` as the guard.

### 3. `src/model.ts` — six changes

#### 3a. `CascadeOp` union (current line ~62) — add `arm` op

```ts
| { kind: 'arm'; handlerLoop: string; reason: string } // arm a handler loop on invalidation of L
```

The `arm` op carries the handler loop name (not a single path) because the handler may produce multiple outputs, and materializing them belongs in the op-applier which has access to the full def.

#### 3b. `resolvedEffect` (current lines ~146–150) — widen return type

```ts
function resolvedEffect(loop: LoopDef | undefined): { idempotent: boolean; onInvalidate: 'pin' | 'escalate' | string } {
  const idempotent = loop?.effect?.idempotent ?? true;
  const onInvalidate = loop?.effect?.onInvalidate ?? 'escalate';
  return { idempotent, onInvalidate };
}
```

#### 3c. `pendingOwed` (current lines ~252–289) — D-A: exclude handler outputs

A loop H is a handler iff some other loop L declares `effect.onInvalidate === H.name`. Handler outputs MUST NOT be seeded owed at instance creation — they are dormant by construction and only armed on demand.

Before the main loop, build the handler loop name set:
```ts
const handlerLoopNames = new Set<string>();
for (const loop of def.loops) {
  const oi = loop.effect?.onInvalidate;
  if (oi && oi !== 'pin' && oi !== 'escalate') handlerLoopNames.add(oi);
}
```

Then inside the for-loop, before calling `ensure`:
```ts
for (const loop of def.loops) {
  if (handlerLoopNames.has(loop.name)) continue; // handler: dormant, skip normal seeding
  // ... existing mode === 'map' branch and else branch unchanged ...
}
```

This is the complete dormancy gate (D-A). Handler outputs are not seeded, not debts, not visible to `eligibleFirings` at creation.

#### 3d. `maintainDecisions` invalidation routing (current lines ~474–489) — add handler branch (D-B, D-C)

Current `else` branch handles escalate. Add a named-handler branch before it:

```ts
if (effect.idempotent) {
  ops.push({ kind: 'reject', path: art.path, reason: `auto-invalidated: ${moved ?? 'an input'} moved version` });
} else if (effect.onInvalidate === 'pin') {
  ops.push({ kind: 'pin', path: art.path, reason: `pinned: ${moved ?? 'an input'} moved; held green per effect.onInvalidate=pin` });
} else if (effect.onInvalidate !== 'escalate') {
  // Named handler (D-B): pin the original L + arm the handler H.
  // Pin re-points fingerprint so the next pass sees invariant holds (no thrash, D-C).
  ops.push({ kind: 'pin', path: art.path, reason: `pinned: ${moved ?? 'an input'} moved; original kept green, handler '${effect.onInvalidate}' armed` });
  ops.push({ kind: 'arm', handlerLoop: effect.onInvalidate, reason: `handler '${effect.onInvalidate}' armed: L='${art.producer}' artifact pinned on input move` });
} else {
  ops.push({ kind: 'reject', path: art.path, held: true, reason: `held: ${moved ?? 'an input'} moved; irreversible — escalate to human (effect.onInvalidate=escalate)` });
}
```

Key: the `pin` op for L is pushed first; `arm` is pushed second. `settle()` in engine.ts processes ops in order and re-reads artMap per iteration, so both reach the store. The pin re-points L's fingerprint, so the NEXT iteration of `maintainDecisions` sees `fingerprintMatches` return true for L — no second pin is emitted (D-C no-thrash). The `arm` op materializes H's outputs as owed; `eligibleFirings` then sees H's owed outputs with green inputs.

#### 3e. `applyOpInMemory` (current lines ~1148–1209) — handle `arm` op

Insert before the final reject/retract block:
```ts
if (op.kind === 'arm') {
  // Arm the handler: materialize each of its singleton/collection outputs as owed
  // if absent, or re-arm to owed if currently green (re-invalidation, D-C re-arm).
  const handlerLoop = def.loops.find((l) => l.name === op.handlerLoop);
  if (!handlerLoop) return;
  for (const p of handlerLoop.produces.filter((pp) => pp.kind === 'singleton')) {
    const existing = arts.get(p.stem);
    if (!existing) {
      arts.set(p.stem, {
        workflow: '',
        path: p.stem,
        producer: handlerLoop.name,
        acceptance: 'owed',
        version: 0,
        reasons: [],
        judgmentRejects: 0,
        schemaRejects: 0,
      });
    } else if (existing.acceptance === 'green') {
      // Re-invalidation: H already fired once; re-arm it.
      arts.set(p.stem, { ...existing, acceptance: 'owed' });
    }
    // If owed or rejected already, no change needed (already a debt).
  }
  // Collection seals: same pattern on sealPath(stem).
  for (const p of handlerLoop.produces.filter((pp) => pp.kind === 'collection')) {
    const sealKey = p.stem + '.sealed';
    const existing = arts.get(sealKey);
    if (!existing) {
      arts.set(sealKey, {
        workflow: '',
        path: sealKey,
        producer: handlerLoop.name,
        acceptance: 'owed',
        version: 0,
        reasons: [],
        judgmentRejects: 0,
        schemaRejects: 0,
        sealOf: p.stem,
      });
    } else if (existing.acceptance === 'green') {
      arts.set(sealKey, { ...existing, acceptance: 'owed' });
    }
  }
  return;
}
```

Note: `sealPath` in model.ts constructs `stem + '.sealed'`. Use the inline equivalent `p.stem + '.sealed'` to avoid importing. Check actual `sealPath` implementation — it is `(stem: string) => stem + '.sealed'` (confirmed at model.ts sealPath usages) — so the inline is correct.

### 4. `src/engine.ts` — `applyOp` handler for `arm` op

In `applyOp` (current lines ~819–862), insert a new branch before the final reject/retract block, mirroring `applyOpInMemory` but writing to `this.store.putArtifact` with `workflow` populated:

```ts

  const handlerLoop = def.loops.find((l) => l.name === op.handlerLoop);
  if (!handlerLoop) return;
  const now = Date.now();
  // Singleton outputs
  for (const p of handlerLoop.produces.filter((pp) => pp.kind === 'singleton')) {
    const existing = arts.get(p.stem);
    if (!existing) {
      this.store.putArtifact({
        workflow,
        path: p.stem,
        producer: handlerLoop.name,

        version: 0,

        judgmentRejects: 0,
        schemaRejects: 0,

      // Re-arm: H fired before; re-invalidation re-arms it.

        ...existing,
        acceptance: 'owed',
        reasons: [...existing.reasons, reason('reopen', 'structural', 'engine', op.reason, existing.version)],
      });
    }
    // owed/rejected: already a debt, no change.
  }
  // Collection seals
  for (const p of handlerLoop.produces.filter((pp) => pp.kind === 'collection')) {
    const sealKey = p.stem + '.sealed';
    const existing = arts.get(sealKey);
    if (!existing) {
      this.store.putArtifact({
        workflow,
        path: sealKey,
        producer: handlerLoop.name,
        acceptance: 'owed',
        version: 0,
        reasons: [reason('reopen', 'structural', 'engine', op.reason, 0)],
        judgmentRejects: 0,
        schemaRejects: 0,
        sealOf: p.stem,
      });
    } else if (existing.acceptance === 'green') {
      this.store.putArtifact({
        ...existing,
        acceptance: 'owed',
        reasons: [...existing.reasons, reason('reopen', 'structural', 'engine', op.reason, existing.version)],
      });
    }
  }
  return;

```

The `reason` helper is already used throughout `engine.ts`'s `applyOp` — use it as-is. `arts` is the snapshot passed into `applyOp`; `this.store.putArtifact` writes to the store. The next `settle()` iteration re-reads artMap, so H's new owed outputs are visible to `pendingOwed` and `maintainDecisions` on the next pass.

**Important**: `settle()` calls `pendingOwed` each iteration before `maintainDecisions`. With the dormancy guard in place, `pendingOwed` will NOT seed H's outputs at instance creation. But after an `arm` op writes H's outputs as owed via `this.store.putArtifact`, the NEXT iteration's `pendingOwed` call sees `arts.has(path)` return true and skips re-seeding — correct, no duplication.

## Edge cases

1. **Handler loop does not exist at runtime**: `validateDef` rejects the def before any instance is created; `loopByName` returning undefined inside `applyOp`/`applyOpInMemory` is a belt-and-suspenders no-op guard.

2. **Self-handler**: caught by D-D in `validateDef` (`oi === l.name`). At runtime L would pin itself and arm itself, creating infinite cascade — blocked at def time.

3. **Handler produces nothing**: caught by D-D (`handlerLoop.produces.length === 0`). Without outputs, `arm` would push no artifacts to the store — no debt, no eligibility. Hard error.

4. **No-thrash (D-C)**: The `pin` op re-points L's fingerprint to current input versions. On the very next `maintainDecisions` pass, `fingerprintMatches` returns true for L → no new pin, no new arm. `settle()` converges in at most 2 iterations (iter 1: pin+arm; iter 2: verify stable).

5. **Re-invalidation (D-C re-arm)**: If input moves AGAIN after H has greened, L's new fingerprint mismatches → `pin` L again + `arm` H again. The arm op finds H's output green → re-arms to owed. H re-fires. This is the intended re-invalidation loop.

6. **H is already owed/rejected**: The arm op's `else if (existing.acceptance === 'green')` branch is the only mutating branch. If H is already owed or rejected (already a debt), no change — idempotent.

7. **H is a map producer**: Map loops produce per-element outputs; their seals are the armable unit. The arm op targets `kind === 'collection'` seals. Map elements become owed dynamically (when their bound element greens), exactly as today — the arm op does not need to enumerate map children.

8. **Terminal L**: `art.terminal` guard at line ~463 (`if (art.acceptance === 'green' && !art.terminal)`) prevents terminal artifacts from entering the routing branch at all — no change needed.

9. **H is itself non-idempotent**: H's own effect contract is independent. When H greens, it may itself have `effect:` routing; the engine handles that on H's next invalidation normally. No special case.

10. **`pendingOwed` idempotency after arm**: After arm writes H's artifact to the store, subsequent `pendingOwed` calls skip it (the `arts.has(path)` guard). No double-seeding.

### `docs/design.md` — extend §20.6

The existing §20.6 reads: "named-handler routing is a planned follow-up; any non-pin/escalate string is currently a hard validateDef error."

Replace with a full named-handler subsection covering:
- What `onInvalidate: <loopName>` means: pin original (L stays green, fingerprint re-pointed) + arm handler (H's outputs materialized as owed on demand).
