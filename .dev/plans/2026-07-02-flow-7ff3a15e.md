# Plan: untrack three stale .dev/plans/*.md files

## Goal

Pure hygiene, no functional change. .gitignore has `.dev/*` with only
`!.dev/checks.sh` excepted, but three plan snapshots are tracked on
origin/main anyway because .gitignore never retroactively untracks files
that were already force-added/committed (they landed in commit 0c16704,
PR #55). This is a recurring incident -- same root cause as PRs #38 and
#47, already recorded in the knowledge graph (node "gitignore does not
un-track already-committed files"). Fix: `git rm --cached` exactly the
three offending files so `git ls-files .dev/` shows only checks.sh,
matching the .gitignore's existing intent. Files stay on disk (they are
live dev-local plan snapshots for other in-flight runs) -- only the git
index entry is removed.

## Files to touch

Untrack (do not delete from disk):
- .dev/plans/2026-07-02-flow-01ebf9db.md
- .dev/plans/2026-07-02-flow-ddc6ddd0.md
- .dev/plans/2026-07-02-flow-f49660b9.md

No other file changes. Confirmed via `git grep` across the tracked tree
that no code, docs, or config reference these three filenames -- nothing
else needs to change, and .gitignore itself is already correct and needs
no edit.

## Approach

1. `git rm --cached .dev/plans/2026-07-02-flow-01ebf9db.md .dev/plans/2026-07-02-flow-ddc6ddd0.md .dev/plans/2026-07-02-flow-f49660b9.md`
   -- removes the index entries only (no `-f`, no `-r`; these are three
   plain tracked files, not a directory). Do NOT use plain `git rm`
   (without `--cached`) -- that would delete the files from disk too, and
   they are live dev-local artifacts other runs may still read.
2. Confirm the files remain on disk (`ls .dev/plans/`) and that git now
   treats them as ignored (`.gitignore`'s existing `.dev/*` /
   `!.dev/checks.sh` rules take over automatically once untracked -- no
   .gitignore edit needed).
3. Commit with a message naming the hygiene fix and the recurring pattern,
   e.g.: "chore: untrack stale .dev/plans/*.md snapshots (recurs per PR
   #38, #47, #55)".
4. Open the PR.

Do not touch .gitignore -- it already expresses the right intent. Do not
add a pre-commit hook or CI guard to prevent recurrence; that is scope
creep beyond this proposal, which explicitly says "do NOT touch anything
else." A permanent prevention mechanism (if wanted) is a separate future
proposal.

## Edge cases

- None of the three files are referenced elsewhere in the tracked tree
  (verified with `git grep -n "<each filename>"` across the repo -- zero
  hits), so untracking them cannot break a build, doc link, or test
  fixture path.
- `git rm --cached` on a file whose index and working-tree copies are
  identical is a no-op on disk; after running it, `git status` should show
  the three files as newly untracked-then-ignored, not as deletions of
  content.
- Because `.dev/*` is already ignored, after this commit lands a clean
  `git status` must NOT list these files at all (they're ignored, not just
  untracked-and-visible) -- double check with `git status --ignored` if
  unsure.

## Verification

- `git ls-files .dev/` -> must print exactly `.dev/checks.sh`.
- `git status` -> the three files must not appear.
- `bash .dev/checks.sh` (i.e. `npm run check`) -> must stay green; nothing
  in the test/build/lint pipeline depends on these files being tracked,
  since they are dev-local plan snapshots, not source or fixtures.
- `git diff --cached --stat` -> should show exactly three deletions from
  the index, zero insertions, zero other files touched.

## Docs

None needed. This is pure repo hygiene with no user-facing or API-surface
change, and the proposal explicitly prefers zero doc changes. .gitignore
already documents the intent via its existing comment ("# dev
delivery-system scaffolding (local foreman state -- not part of the
library)") and needs no edit.
