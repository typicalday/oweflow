# Plan: workflow-level `outputs:` field + dead-end lint exemption

## Goal

Add an optional workflow-level `outputs: [stem, ...]` field to the oweflow def language. This is the first and smallest slice of the workflow-composition work (design doc ¬ß5). It:
1. Declares the workflow's intentional public leaves / embedding interface.
2. Hard-errors in `validateDef` when an `outputs:` entry names a stem that no loop produces.
3. Adds `outputs:` as a **third** exemption in `deadEndWarnings`, additive to the two existing exemptions (`terminal:` and `generates:`). Neither existing exemption is touched.
4. Does **NOT** wire `outputs:` to `include:`/`calls:`; does NOT add `effect:`; does NOT change `model.ts`, `engine.ts`, or any cascade logic.

Baseline: all 306 existing tests pass (`npm run check`). Back-compat is sacred ‚Äî every existing def, example, and test must pass untouched.

## Approach

### Step 1 ‚Äî `src/types.ts`: add field to `WorkflowDef`

Locate the `WorkflowDef` interface (currently around line 167). Add `outputs?: string[]` after `loops` and before `dir`. Mirror the JSDoc style of `generates?` on `LoopDef`:

```ts
/** Workflow-level public outputs / embedding interface (design doc ¬ß5.2).
 *  Declared stems are intentional leaves: lint-exempt from dead-end warnings.
 *  A stem listed here that no loop produces is a hard validateDef error. */
outputs?: string[];
```

### Step 2 ‚Äî `src/defs.ts`: RawDef interface

Locate `interface RawDef` (around line 60). Add `outputs?: unknown;` alongside the existing optional fields (`title`, `description`, `inputs`, `loops`, `invariants`). Keep the same style.

### Step 3 ‚Äî `src/defs.ts`: parse in `buildDef`

Locate `buildDef` (around line 226). After the block that builds `invariants` and before `return def`, add parsing for `outputs`:

```ts
if (r.outputs !== undefined) {
  const outs = asStringArray(r.outputs, 'outputs');
  if (outs.length > 0) def.outputs = outs;
}
```

Use the existing `asStringArray` helper ‚Äî it throws `DefError` on non-string elements. Empty array means omit the field (mirrors how `generates:` is omitted when length === 0).

**Important:** parse at the WORKFLOW level inside `buildDef`, in the same block that handles `title`/`description`/`invariants` ‚Äî NOT inside `buildLoop`. The field is a sibling of `loops:` and `name:`.

### Step 4 ‚Äî `src/defs.ts`: hard-error in `validateDef`

Locate `validateDef` (around line 308). Add a new validation block AFTER the existing `produces: and generates:` overlap check (around line 358) and BEFORE the reachability/cycle checks. The check:

1. Build the set of all stems produced by any loop by iterating `def.loops` and collecting `p.stem` for all `l.produces`. Because `generates:` patterns are unioned into `l.produces` at build time, `generates:` stems are already in this set ‚Äî an `outputs:` entry naming a `generates:` stem is valid without any special handling.
2. For each stem in `def.outputs` (if any), push an error if it is not in the all-produced-stems set:

```ts
if (def.outputs && def.outputs.length > 0) {
  const allProducedStems = new Set<string>(
    def.loops.flatMap((l) => l.produces.map((p) => p.stem)),
  );
  for (const stem of def.outputs) {
    if (!allProducedStems.has(stem)) {
      errors.push(`outputs: entry '${stem}' is not produced by any loop`);
    }
  }
}
```

Do NOT add input names to `allProducedStems`. The proposal says `outputs:` entries must name stems a loop PRODUCES. Input names are not produced by loops, so they should be an error.

Place this block using the same `errors.push`/accumulation style as the `generates:` overlap check immediately above it.

### Step 5 ‚Äî `src/defs.ts`: third exemption in `deadEndWarnings`

Locate `deadEndWarnings` (around line 524). Currently it has two exemptions:
- `if (l.terminal) continue;` ‚Äî whole-loop early-out
- `if (generatedStems.has(p.stem)) continue;` ‚Äî generates: exemption

Add workflow `outputs:` as a Set built before the loop body:

```ts
// stems declared in workflow outputs: are intentional public leaves ‚Äî lint-exempt
const workflowOutputStems = new Set<string>(def.outputs ?? []);
```

Then inside the per-produce inner loop, add a third guard after the existing `generates:` one:
```ts
if (workflowOutputStems.has(p.stem)) continue; // workflow outputs: exempt
```

Guard order inside the inner loop: (1) `p.kind === 'map'` skip, (2) `generatedStems` skip, (3) `workflowOutputStems` skip, (4) warn. Do not reorder or remove the existing guards.

Also update the warning message to mention `outputs:` as a third remedy:

```
`loop '${l.name}' produces '${p.stem}' but nothing consumes it ` +
`(dead-end output; declare it under generates: if no consumer is expected, ` +
`list it in the workflow outputs: if it is a public interface leaf, ` +
`or mark the loop terminal: true if this is an intended sink)`
```

## Edge cases

1. **`outputs:` names a `generates:` stem** ‚Äî VALID. Because `generates:` patterns are unioned into `l.produces` at build time, the stem is in `allProducedStems`. No special-case code needed.
2. **`outputs:` names an input stem** ‚Äî treated as an error (not produced by any loop). Intentional per the proposal.
3. **`outputs:` is an empty array** ‚Äî field omitted from `def` (`def.outputs` stays undefined). No validation errors, no lint effect.
4. **`outputs:` names a map-kind stem** ‚Äî the stem is in `allProducedStems`, so no hard error. In `deadEndWarnings`, map outputs are skipped by the `p.kind === 'map'` check before the `workflowOutputStems` check, so the exemption is never reached for map stems ‚Äî harmless.
5. **Existing defs with no `outputs:` key** ‚Äî `def.outputs` is undefined; `def.outputs ?? []` is `[]`; new Set is empty; all existing behavior unchanged.
6. **`outputs:` names a stem consumed further downstream** ‚Äî valid, no error. The validation only checks whether the stem is produced by some loop.
7. **Duplicate entries in `outputs:`** ‚Äî `asStringArray` returns them; Set deduplicates for lookup; no duplicate-detection check needed (not requested).

**Test (b)** ‚Äî unlisted unconsumed leaf STILL warns:

**Test (c)** ‚Äî back-compat: `terminal:` and `generates:` still exempt (all three coexist):

**Test (d)** ‚Äî `outputs:` entry naming a stem no loop produces ‚Üí hard validateDef error:

**Test (e)** ‚Äî `outputs:` entry naming a `generates:` stem ‚Üí VALID, no error, no warning:

- `lintDef(buildDef(...))` returns no warnings for `audit_log`.

## Documentation

### `docs/design.md` ‚Äî new section after ¬ß16

Insert after the `## ¬ß16 Generated outputs` section. Decide at build time whether to renumber ¬ß17‚Üí¬ß18 (cleaner but more diff) or use `## ¬ß16a` (zero renumbering, less diff noise). Prefer inserting as a new numbered section and updating the two cross-reference lines that mention ¬ß17 and ¬ß18. Content:

```
## ¬ß17 Workflow outputs (`outputs:`)

A workflow may declare its public output stems ‚Äî the leaves it intentionally produces as
its embedding interface ‚Äî under a top-level `outputs:` field.

- **Lint exemption:** stems listed in `outputs:` are exempt from `deadEndWarnings`, as a
  third exemption alongside `terminal:` (loop-level) and `generates:` (loop-level). A
  declared public output is self-evidently an intentional leaf.
- **Re-armability:** unlike `terminal: true`, listing a stem in `outputs:` does NOT freeze
  re-arm. The cascade may re-arm an `outputs:`-listed artifact if its upstream inputs move.
- **Validation:** `validateDef` hard-errors if any `outputs:` entry names a stem that no
  loop produces. Stems declared under `generates:` are unioned into `produces` at build
  time and therefore count as produced ‚Äî naming them in `outputs:` is valid.
- **Future use:** `outputs:` will become the boundary contract for workflow composition
  (`include:` / `calls:`). This wiring is not implemented yet.

Relationship of the three exemption mechanisms:

| key | level | lint-exempt | re-armable | primary purpose |
|---|---|---|---|---|
| `terminal: true` | loop | yes | no | destructive completion; green never re-armed |
| `generates:` | loop | yes | yes | internal intentional sink, not the public interface |
| `outputs:` | workflow | yes | yes | public interface / future composition boundary |
```

```
### `outputs:` ‚Äî the workflow interface

A top-level `outputs:` field declares which stems are this workflow's intentional public
leaves ‚Äî the artifacts a parent workflow will later consume when this one is embedded
(`include:`/`calls:`, future feature). A stem listed in `outputs:` is exempt from
dead-end warnings (a third exemption alongside `generates:` and `terminal: true`), but
unlike `terminal:` it stays re-armable.

| key | level | lint-exempt | re-armable | meaning |
|---|---|---|---|---|
| `terminal: true` | loop | yes | **no** | destructive completion; never re-armed |
| `generates:` | loop | yes | yes | internal intentional sink, not the public interface |
| `outputs:` | workflow | yes | yes | public interface / future composition boundary |

Validation: every stem in `outputs:` must be produced by at least one loop (stems under
`generates:` count, since they are unioned into `produces` at build time).
```
