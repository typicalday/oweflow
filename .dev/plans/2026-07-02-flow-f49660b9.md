# Instance-to-definition pinning (snapshot + drift flag, no refuse-to-advance)

## Goal

Today `Engine.createInstance` stores only the def **name** on the `workflow`
row (`src/store.ts` `WorkflowData.def`), and every def-touching engine verb
re-resolves the def by name through the single chokepoint `Engine.defFor`
(`src/engine.ts:1548`, ~13 call sites: `provideInput`, `tick`, `green`, `emit`,
`seal`, `reject`, `retract`, `skip`, `retry`, `reap`, `status`, `nextAlarm`,
`fireSettled`). Editing the source YAML mid-flight silently rewires every
running instance of that def name to the new shape on its very next verb call
— no detection, no warning.

This delivery pins each instance to the compiled def it was created against:

1. Snapshot the compiled def (JSON) + a content hash on the workflow row at
   `createInstance` time.
2. The engine resolves from that snapshot when present; falls back to today's
   name-resolution for pre-existing rows that have no snapshot (backward
   compatible, not an error).
3. `status` surfaces `defDrift: true` when the currently-loaded def's hash
   differs from the pinned snapshot's hash — informational only, the engine
   keeps running off the pinned snapshot regardless.
4. A new CLI command `owenloop adopt <wf>` re-pins the instance to the
   CURRENT def (re-snapshot + re-hash) and runs `settle()` so any new debts
   introduced by the updated shape materialize immediately, as a deliberate
   operator action.

Explicitly OUT OF SCOPE: no throwing/refuse-to-advance behavior anywhere.
That was a different, previously-explored design (hash-only, `DefDriftError`)
for the same problem, abandoned — not what this builds. This feature must
stay additive/backward-compatible end to end: old rows with no snapshot keep
working exactly as they do today.

There is no prior-plan artifact to inherit (the earlier planning instance for
this feature, wf_8b0632b9ecaafa2a374308a8, was deleted; its research is not
recoverable from `dev graph owenloop search` — confirmed empty on several
query attempts). This plan was produced by reading the current codebase
directly — grounded in what's actually on `main` today, not a memory of the
old design.

## 1. Store: schema + migration (`src/store.ts`)

### Schema version

Current `SCHEMA_VERSION = '5'` (`src/store.ts:141`). Bump to `'6'`. The
`StoreVersionError` guard (`src/store.ts:322-333`, from PR #48) reads:

```ts
if (cur !== undefined) {
  const curNum = parseInt(cur, 10);
  const ownNum = parseInt(SCHEMA_VERSION, 10);
  if (curNum > ownNum) throw new StoreVersionError(...);
}
if (cur !== SCHEMA_VERSION) this.setMeta('schema_version', SCHEMA_VERSION);
```

This logic is generic over the version string — it does not need to change.
Bumping `SCHEMA_VERSION` to `'6'` automatically:
- keeps "opening a DB already at current SCHEMA_VERSION is a no-op" true (now
  compares against '6'),
- keeps "older on-disk version upgrades in place" true (a stamped '5' or '3'
  is `< 6`, upgrades silently via `migrate()` + `setMeta`),
- keeps "newer-than-binary on-disk version throws and does not rewrite" true
  (a stamped '7' is `> 6`, still throws `StoreVersionError`, unchanged code
  path).

Do not touch the guard's comparison logic — only the constant. Update its
doc comment (`src/store.ts:132-140`) to mention this migration as the reason
for the bump, same style as the existing comment references "the next
additive step."

## 2. Hash + snapshot helper (`src/defs.ts`)

Add a small, pure helper next to `buildDef` (consistent with where
`WorkflowDef`-shaped logic already lives), exported from `src/index.ts`
alongside `buildDef`/`parseDef`/`loadDefs`:

```ts
/**
 * Instance-to-definition pinning (§28): a stable content hash of a compiled
 * WorkflowDef, used to detect when the live definition has drifted from an
 * instance's pinned snapshot. Sha256 of the def's canonical JSON form,
 * truncated to 16 hex chars — long enough to be practically collision-free
 * for this purpose (detecting accidental drift, not an adversarial actor),
 * short enough to be a legible field in `status` output.
 */
export function hashDef(def: WorkflowDef): string {
  return createHash('sha256').update(JSON.stringify(def)).digest('hex').slice(0, 16);
}
```

Notes:
- Check whether `defs.ts` already imports from `node:crypto`; if not, add
  `import { createHash } from 'node:crypto';` there. This mirrors the
  existing `detId`/fingerprint hashing precedent in `src/util.ts:6-9` and
  `src/model.ts`'s `computeFingerprint` — deterministic hashing of a
  fixed-shape object is the established idiom here, not a new one.
- `JSON.stringify` on a plain object is stable here because `buildDef`/
  `parseDef`/`expandIncludes` always construct the def object with the same
  field order — no dynamic key ordering or prototype tricks. Good enough for
  drift DETECTION (the goal), not intended as a cryptographic content
  address.
- Reuse `hashDef` for stamping AND for the drift comparison — never
  recompute the hash a different way in two places.
- The "snapshot" persisted is the def AS RETURNED BY `resolveDef` at
  `createInstance` time — i.e., the fully expanded, already-`buildDef`'d /
  `validateDef`'d object with `_includes` already stripped (see
  `expandIncludes`, `src/defs.ts:644-671`, and `loadDefs`'s two-phase build).
  This is the "compiled def" the proposal specifies, not the raw YAML.
  `loadDefs` runs `expandIncludes` before handing back its map, and
  `createEngine({ defs })`'s in-memory path is expected pre-expanded too, so
  `resolveDef(name)` should never hand back an object still carrying
  `_includes` in practice — confirm this holds when implementing (e.g. via a
  quick assertion or by checking `factory.ts`/`cli.ts`'s resolver
  construction) rather than assuming silently.

## 4. Engine resolution: fallback (`defFor`, `src/engine.ts:1548-1552`)

Current:

```ts
private defFor(workflow: string): WorkflowDef {
  const wf = this.store.getWorkflow(workflow);
  if (!wf) throw new Error(`no such workflow instance: ${workflow}`);
  return this.resolveDef(wf.def);
}
```

New:

```ts
private defFor(workflow: string): WorkflowDef {
  const wf = this.store.getWorkflow(workflow);
  if (!wf) throw new Error(`no such workflow instance: ${workflow}`);
  // Instance-to-definition pinning (§28): prefer the pinned snapshot taken at
  // create time (or last `adopt`). Rows created before this feature shipped
  // have no snapshot — fall back to today's name-resolution, unchanged. This
  // is the compatibility path, not an error: an un-pinned instance behaves
  // exactly as it always has.
  if (wf.defSnapshot !== undefined) return wf.defSnapshot;
  return this.resolveDef(wf.def);
}
```

This is the ONLY change needed to make all ~13 call sites through `defFor`
resolve from the pin — it is the single chokepoint the proposal identifies,
so no other call site needs touching. Verify by re-grepping
`this.defFor(workflow)` after the change (already enumerated above: lines
265, 434, 655, 819, 899, 963, 1023, 1042, 1073, 1139 (via `def ?? this.defFor`),
1166, 1207, 1255 — exact line numbers will shift slightly as code is added,
re-grep rather than trust these).

Two things to double check as the "calls:/include: bypass" the proposal
flags:
- `maintainCalls` doesn't call `resolveDef` itself but is always invoked with
  a `def` already resolved by its caller (`tick`, `provideInput`) — those
  callers already go through `defFor`, so `maintainCalls` inherits the pin
  transitively. No separate fix needed there.
- Confirm there is no OTHER direct `this.resolveDef(...)` call in

## 5. `defDrift` status flag

### Where it lives

Compute `defDrift` in `Engine.status()` (`src/engine.ts:1165-1186`) rather
than inside the pure `workflowStatus` in `model.ts`. Rationale: drift
detection needs BOTH the pinned snapshot (from the store row) AND the
CURRENTLY-LOADED live def (from `this.resolveDef(wf.def)`, i.e. what's on
disk/in-memory right now) — that's an engine-level concern (it has the store
AND the resolver), not a pure function of `(def, arts)` like everything else
in `model.ts`. Keep `model.ts`/`WorkflowStatus` untouched; add the field as
an engine-level enrichment, the same way `status()` already enriches
`debts[].failedRuns`/`attempts` post-hoc (lines 1174-1184 — same pattern,
just one more top-level field instead of per-debt).

Add a new exported type in `engine.ts` (it's the engine-level status shape,
so it belongs where `TickResult`/`CommitResult` live, not in `model.ts`):

```ts
export interface EngineWorkflowStatus extends WorkflowStatus {
  /** §28: true when the def currently loaded for this workflow's name differs
   *  (by content hash) from the snapshot this instance is pinned to. Present
   *  (true/false) whenever the instance has a pinned snapshot; absent for a
   *  legacy un-pinned instance (nothing to compare) or when the live def
   *  can't currently be resolved at all (can't determine drift, not "no
   *  drift"). Informational — the engine keeps operating from the pinned
   *  snapshot regardless; clear the drift by running `owenloop adopt <wf>`. */
  defDrift?: boolean;
}
```

Change `Engine.status`'s return type from `WorkflowStatus` to
`EngineWorkflowStatus`, and add the computation:

```ts
status(workflow: string): EngineWorkflowStatus {
  const wf = this.store.getWorkflow(workflow);
  if (!wf) throw new Error(`no such workflow instance: ${workflow}`);
  const def = this.defFor(workflow);
  const arts = this.artMap(workflow);
  const st: EngineWorkflowStatus = workflowStatus(def, arts);
  // ... existing debts enrichment loop unchanged ...
  if (wf.defSnapshot !== undefined) {
    let live: WorkflowDef | undefined;
    try { live = this.resolveDef(wf.def); } catch { live = undefined; }
    if (live !== undefined) {
      const pinnedHash = wf.defHash ?? hashDef(wf.defSnapshot);
      st.defDrift = hashDef(live) !== pinnedHash;
    }
  }
  return st;
}
```

Notes:
- `defFor(workflow)` is called again here for `def` (unchanged from today);
  it now transparently returns the pin per Section 4. `this.resolveDef` is
  called SEPARATELY and directly for `live`, deliberately bypassing the pin.
- Wrap `this.resolveDef(wf.def)` in try/catch: the live def might have been
  deleted/renamed on disk since the instance was pinned (a real scenario now
  that pinning means the instance no longer NEEDS the live def to keep
  running). Resolution failure here must NOT throw `status` — that would
  regress today's `status` contract for pinned instances that no longer have
  a resolvable live name. Treat "can't resolve live def" as "can't determine
  drift" (leave `defDrift` unset) rather than an error. This is an important
  edge case to test explicitly (Section 7).
- Prefer `wf.defHash` (stamped once, cheap to compare) over re-hashing
  `wf.defSnapshot` every call, falling back to hashing the snapshot only if
  `defHash` is ever missing while `defSnapshot` is present (shouldn't happen
  given Section 3 always stamps both together — defensive only).
- `status --all` (`src/cli.ts:391-403`, `statusEntry` at `src/cli.ts:656-676`)
  and `list`'s `safeStatus` (`src/cli.ts:646-654`) call `engine.status`
  indirectly. Check `statusEntry`'s exact shape — it likely spreads
  `...engine.status(w.id)` — to confirm `defDrift` flows through
  automatically; if `statusEntry`/`safeStatus` instead destructure specific
  fields rather than spreading the whole object, add `defDrift` explicitly
  there too.

## 6. CLI `owenloop adopt <wf>`

### Engine method

Add a public method on `Engine`:

```ts
/**
 * §28: re-pin `workflow` to the CURRENTLY-LOADED def for its name — re-
 * snapshot + re-hash, overwriting the stored pin — then settle() so any new
 * debts introduced by the updated shape (e.g. a newly added required input,
 * a renamed producer) materialize immediately as a deliberate act, not a
 * surprise on the next unrelated verb call. This is the explicit opt-in to
 * re-wire an in-flight instance onto new def content; contrast defFor's
 * silent, unconditional pin-preservation.
 */
adopt(workflow: string): { workflow: string; defHash: string; previousHash?: string } {
  const wf = this.store.getWorkflow(workflow);
  if (!wf) throw new Error(`no such workflow instance: ${workflow}`);
  const freshDef = this.resolveDef(wf.def);
  const newHash = hashDef(freshDef);
  const previousHash = wf.defHash;
  this.store.tx(() => {
    this.store.repinWorkflowDef(workflow, freshDef, newHash);
    this.settle(workflow, freshDef);
  });
  this.fireSettled(workflow);
  return previousHash !== undefined
    ? { workflow, defHash: newHash, previousHash }
    : { workflow, defHash: newHash };
}
```

Notes:
- `this.resolveDef(wf.def)` here is intentional and correct — `adopt` is
  precisely the one place that MUST read the live current def, bypassing the
  pin, because re-pinning IS the point.
- Mirrors `provideInput`'s tx/settle/fire pattern (`src/engine.ts:264-293`).
- If `wf.def` no longer resolves (YAML deleted, renamed, syntax error), let
  `resolveDef` throw naturally — `adopt` on an instance whose def is gone is
  a genuine error, unlike `status`'s drift check which tolerates it.
- Do not add a new `EngineEvent` variant for this unless it turns out to be
  needed for a test or an embedder use — `fireSettled` already notifies
  observers of the resulting settle, and `adopt` isn't a producer-verb
  commit. Avoid speculative surface area.

## Key assumptions made (flagged, not escalated — routine, reversible)

- `hashDef` lives in `src/defs.ts` and is exported from `src/index.ts`; an
  equally valid choice is keeping it engine-internal/unexported. Low cost to
  change either way — not a one-way door.
- `defDrift` defaults to `false` (not absent) whenever a snapshot exists and
  currently matches; absent only for legacy un-pinned instances, or when the
  live def can't currently be resolved at all. Minor API-shape choice, easy
  to adjust before merge if the builder finds a cleaner contract.
- `adopt`'s JSON return shape (`{ ok, workflow, defHash, previousHash? }`) is
  a reasonable default; the CLI/engine boundary here is thin and easy to
  adjust in review if a different shape reads better.
- New schema columns are named `def_snapshot`/`def_hash` (snake_case, matching
  every other column in the `workflow` table) and their TS-side names are
  `defSnapshot`/`defHash` (camelCase, matching every other field) — this is
  just following the file's 100%-consistent existing convention, not a new
  decision.

None of these rise to a one-way-door architectural choice — they're the kind
of ordinary implementation-shape gaps this plan is expected to fill
autonomously.
