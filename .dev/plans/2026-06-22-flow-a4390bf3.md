# Plan: Silent-false-green CLI fix (src/cli.ts sync)

## Goal

Bring `liveloop/main/src/cli.ts` into sync with the reference implementation already shipped in `typicalday/dev` PR #63. The bug: `green`, `emit`, and `seal` commands always exit 0 even when the engine returns a non-success outcome (`born-rejected` or `schema-rejected`). Callers read stdout and treat it as success when nothing was actually registered.

## Engine outcome discriminators (verified against src/engine.ts)

- `engine.green()` returns `CommitResult` (src/engine.ts lines 107-113): `outcome: 'green' | 'born-rejected' | 'schema-rejected'`
  - Success discriminator: `outcome === 'green'`
- `engine.emit()` returns `EmitResult` (src/engine.ts lines 116-122): `outcome: 'emitted' | 'born-rejected' | 'schema-rejected'`
  - Success discriminator: `outcome === 'emitted'`
- `engine.seal()` returns `CommitResult` (same type as green, verified at src/engine.ts line 763)
  - Success discriminator: `outcome === 'green'`

These are confirmed from src/engine.ts directly, not assumed from dev PR #63.

## Files to touch

- `src/cli.ts` — only production file that changes.
- `test/cli.test.ts` — new tests added (no new file created).
- `docs/` — scan for exit-code docs; update if found, else green with updated:false.

## Changes to src/cli.ts

### 1. Change dispatch return type from void to number

Line 210 currently reads:
  function dispatch(command: string, io: CliIO, args: Args): void {

Change to:
  function dispatch(command: string, io: CliIO, args: Args): number {

The TypeScript compiler will then enforce that every branch returns an explicit integer. This catches any forgotten conversions at typecheck time.

### 2. Update main() to consume the dispatch return value

Around line 624:
  dispatch(command, io, args);
  return 0;

Change to:
  return dispatch(command, io, args);

### 3. Fix the green case (around line 468)

Change:
  const res = engine.green(wf, run, path, value, { terminal: flag(args, 'terminal') });
  print(io, res);
  return;

To:
  const res = engine.green(wf, run, path, value, { terminal: flag(args, 'terminal') });
  print(io, res);
  if (res.outcome !== 'green') {
    io.err(`green ${path}: ${res.outcome}${res.reason ? ' — ' + res.reason : ''}`);
    return 1;
  }
  return 0;

### 4. Fix the emit case (around line 477)

Change the inline print:
  print(io, engine.emit(wf, run, items));
  return;

To a captured-then-checked pattern:
  const emitRes = engine.emit(wf, run, items);
  print(io, emitRes);
  if (emitRes.outcome !== 'emitted') {
    io.err(`emit: ${emitRes.outcome}${emitRes.reason ? ' — ' + emitRes.reason : ''}`);
    return 1;
  }
  return 0;

### 5. Fix the seal case (around line 492)

Change the inline print:
  print(io, engine.seal(wf, run, parseJson(last(args, 'value'))));
  return;

To a captured-then-checked pattern:
  const sealRes = engine.seal(wf, run, parseJson(last(args, 'value')));
  print(io, sealRes);
  if (sealRes.outcome !== 'green') {
    io.err(`seal ${sealRes.path}: ${sealRes.outcome}${sealRes.reason ? ' — ' + sealRes.reason : ''}`);
    return 1;
  }
  return 0;

### 6. Add a comment to the close case (around line 517)

Directly above the engine.close(...) call, add:
  // close has no outcome discriminator: engine throws on real errors, so {ok:true} is always accurate here.

Leave the close logic unchanged. Change its trailing return; to return 0;

### 7. Convert all other bare return; to return 0;

Every remaining case arm and the early if arms for help, lint, and check end with return; — convert all to return 0; The TypeScript compiler will flag any you miss because dispatch now returns number.

### Test 2: born-rejected green exits non-zero and still prints result JSON

To trigger born-rejected: green a path from run A (success, CAS passes). Then retract that artifact to bump its version. Then tick to get run B. Then call green again from the original (stale) run A — the run A fingerprint no longer matches the current input state, so born-rejected fires.

Alternatively if a simpler trigger exists (e.g., greening the same run+path twice in succession), use that. The exact trigger sequence may require light experimentation; the assertions are fixed:

  test('green: born-rejected exits non-zero and still prints result JSON', () => {
    const { run } = makeCli();
    const wf = run('create', 'delivery', '--provide', `proposal=${J({ text: 'x' })}`).json().workflow;
    const planRun = run('tick', wf).json().orders[0].run;
    // First green succeeds
    assert.equal(run('green', wf, planRun, 'plan', '--value', J({ plan: 'v1' })).code, 0);
    // Retract to move the artifact's version
    run('retract', wf, 'plan', '--by', 'test', '--text', 'forced');
    // Tick so engine sees new owed state; run A is now stale
    run('close', wf, planRun);
    // Green from planRun (stale run) — CAS mismatch => born-rejected
    const r = run('green', wf, planRun, 'plan', '--value', J({ plan: 'v2' }));
    assert.equal(r.code, 1);
    const j = r.json();
    assert.equal(j.outcome, 'born-rejected');
    assert.match(r.err, /born-rejected/);
  });

Note to builder: if the above sequence does not produce born-rejected (because closing the run first is needed or the retract path differs), adjust the trigger sequence until r.json().outcome === 'born-rejected', keeping the assertions identical.

## Edge cases and assumptions

- stdout is unconditional: print(io, res) always runs before the outcome check. The non-zero exit never suppresses JSON stdout. Success-path stdout is byte-for-byte identical to the pre-fix output (only stderr and exit code change on failure).
- Stderr message format: 'green <path>: born-rejected — <reason>' or 'schema-rejected — <reason>'. The reason is omitted when res.reason is undefined. This is human-readable only; it does not affect JSON stdout.
- close is intentionally unchanged: the engine throws on real errors; {ok:true} is always accurate when close returns normally. The added comment documents this intent.
- No async change: dispatch and main are synchronous. Keep them sync. The proposal mentions Promise<number> as dev PR #63's approach (dev may be async for other reasons); this repo's dispatch and main are sync, so return number, not Promise<number>.
- TypeScript imports: CommitResult and EmitResult are not currently imported in src/cli.ts. They do not need to be imported — TypeScript infers the types from the engine method return values. The builder may add the imports for clarity but it is not required.
