# Plan: engine loop-level effect:{idempotent,onInvalidate} + pin/escalate cascade routing

## Goal

Add a loop-level `effect?: { idempotent?: boolean; onInvalidate?: 'escalate' | 'pin' }` field to the workflow definition language and make `maintainDecisions` (src/model.ts) route the forward re-arm cascade on it. Three outcomes:
- **idempotent (default)** — re-arm exactly as today; no behavior change for any existing def.
- **non-idempotent + pin** — the artifact stays green; its fingerprint is re-pointed to current input versions so the cascade sees it as up-to-date.
- **non-idempotent + escalate (default when non-idempotent)** — the artifact is rejected but flagged `held`; the producer is NOT eligible to auto-rebuild; surfaces via the existing status/stall interfaces.

The existing `terminal: true` flag is NOT modified; it remains the legacy spelling, coexisting with `effect:` on the same engine version.

---

### 1. src/types.ts

Add after the `terminal?` JSDoc on `LoopDef`:

```ts
/** Declared per-loop effect contract (design §6.5). Controls forward-cascade routing. */
export interface EffectDef {
  /** If true (default), re-deriving the artifact after inputs move is safe.
   *  When false, the artifact must not silently re-fire. */
  idempotent?: boolean;
  /** Routing when idempotent:false and an input moves.
   *  'pin'      — keep the artifact green, re-point fingerprint to current inputs.
   *  'escalate' — reject the artifact as held; producer not auto-re-eligible.
   *  Defaults to 'escalate' when idempotent:false and omitted. */
  onInvalidate?: 'pin' | 'escalate';
}
```

On `LoopDef`, add between `terminal?` and `body`:
```ts
  /** Loop-level effect contract (§6). Only consulted for non-terminal greens. */
```

```ts
export type RejectKind = 'judgment' | 'structural' | 'validation' | 'invalidated-irreversible';
```

### 2. src/defs.ts

#### 2a. Extend `RawLoop`
Add `effect?: unknown;` to the `RawLoop` interface (after `terminal?`), following the exact pattern of the `generates?` addition in PR #15.

#### 2b. Parse in `buildLoop`
After the `asBool(rl.terminal ...)` block:

```ts
if (rl.effect !== undefined) {
  if (typeof rl.effect !== 'object' || rl.effect === null || Array.isArray(rl.effect)) {
    throw new DefError(`loop '${name}'.effect must be an object`);
  }
  const raw = rl.effect as Record<string, unknown>;
  const effectDef: EffectDef = {};
  if (raw.idempotent !== undefined) {
    effectDef.idempotent = asBool(raw.idempotent, true, `loop '${name}'.effect.idempotent`);
  }
  if (raw.onInvalidate !== undefined) {
    const oi = asString(raw.onInvalidate as unknown, `loop '${name}'.effect.onInvalidate`);
    if (oi !== 'pin' && oi !== 'escalate') {
      // Defer validation error to validateDef below (we push errors, not throw, there)
      // But we still need to carry the raw value for the validator to inspect.
      // Approach: store it verbatim and validateDef will error. So:
      (effectDef as Record<string, unknown>).onInvalidate = oi;
    } else {
      effectDef.onInvalidate = oi;
    }
  }
  loop.effect = effectDef;
}
```

IMPORTANT NOTE: `buildLoop` throws `DefError` on bad shape, but accumulates only in `validateDef`. For the onInvalidate string validation, the cleanest approach — matching the project style — is to allow `asString` to accept the raw string in buildLoop and then push the error in validateDef. Do NOT widen `EffectDef.onInvalidate` to `string` in types.ts; use a type cast/suppression at the parsing site or use a narrow raw type. The alternative (equally acceptable) is to throw in buildLoop and not bother with validateDef: review PR #15/#16's approach — it throws from `parseProduces` on bad shapes, and validateDef only catches wiring-level errors. The builder should follow whichever approach is consistent. Recommendation: parse the raw object in buildLoop, throw DefError immediately for clearly non-string values, and accumulate wiring/semantic errors (unknown onInvalidate string, terminal+effect conflict) in validateDef using `errors.push(...)`. This means the onInvalidate value must be stored as-is through buildLoop for validateDef to check; use a local variable rather than setting `effectDef.onInvalidate` until after validation — or just do an inline check and throw in buildLoop for the invalid-string case (since it is a definitional error, not a wiring error). Either works; prefer throwing in buildLoop for the string-value check to keep it consistent with how `parseDurationSecs` throws on bad cadence strings.

#### 3b. Add a helper to resolve the effective EffectDef for a loop
```ts
/** Resolve the effective effect contract for a loop. Defaults: idempotent=true. */
function resolvedEffect(loop: LoopDef | undefined): { idempotent: boolean; onInvalidate: 'pin' | 'escalate' } {
  const idempotent = loop?.effect?.idempotent ?? true;
  const onInvalidate = loop?.effect?.onInvalidate ?? 'escalate';
  return { idempotent, onInvalidate };
}
```

#### 3c. Modify `maintainDecisions` — the `green && !art.terminal` block (~line 389)

The current block:
```ts

  const versionsOk = fingerprintMatches(arts, req, art.fingerprint ?? {});
  if (!offender && versionsOk) continue; // invariant holds
  if (offender && offenderPath) {
    ops.push(cascadeFromDeadInput(art.path, offender, offenderPath, isMapChild));
    continue;
  }
  const moved = req.find((p) => (arts.get(p)?.version ?? -1) !== (art.fingerprint ?? {})[p]);

```

Become:
```ts
if (art.acceptance === 'green' && !art.terminal) {
  const versionsOk = fingerprintMatches(arts, req, art.fingerprint ?? {});
  if (!offender && versionsOk) continue; // invariant holds

  // Dead/settled input: structural cascade (retract or skip). NOT gated by effect.
  // Only the "changed input" (moved version) path routes on effect.
  if (offender && offenderPath) {
    ops.push(cascadeFromDeadInput(art.path, offender, offenderPath, isMapChild));
    continue;
  }

  // Inputs are all green but at least one moved (version changed). Route on effect.
  const moved = req.find((p) => (arts.get(p)?.version ?? -1) !== (art.fingerprint ?? {})[p]);
  const producerLoop = loopByName(def, art.producer);
  const effect = resolvedEffect(producerLoop);

  if (effect.idempotent) {
    // Default: re-arm as before.
    ops.push({ kind: 'reject', path: art.path, reason: `auto-invalidated: ${moved ?? 'an input'} moved version` });
  } else if (effect.onInvalidate === 'pin') {
    // Pin: keep green, re-point fingerprint to current input versions.
    ops.push({ kind: 'pin', path: art.path, reason: `pinned: ${moved ?? 'an input'} moved; held green per effect.onInvalidate=pin` });
  } else {
    // escalate: reject but mark held — producer must not auto-fire.
    ops.push({
      kind: 'reject',
      path: art.path,
      reason: `held: ${moved ?? 'an input'} moved; irreversible — escalate to human (effect.onInvalidate=escalate)`,
    });
    // The held condition is conveyed through the ReasonEntry kind 'invalidated-irreversible'.
    // applyOpInMemory must append this reason (see 3e).
  }
  continue;
}
```

KEY NOTE: The `cascadeFromDeadInput` path (dead/settled input) is NOT modified — it emits the same retract/skip/reject ops it does today, regardless of effect. This is the distinction the proposal calls for in bullet 5.

#### 3d. Mark escalate ops so applyOpInMemory knows they are held

To carry the 'invalidated-irreversible' marker through to the ReasonEntry, the CascadeOp for the escalate path needs a discriminator. Options:
- Use a new op kind, e.g. `{ kind: 'reject-held'; path; reason }` — cleaner, self-contained.
- Reuse `{ kind: 'reject' }` + a boolean flag `held?: boolean`.

Recommendation: extend the `reject` op with an optional `held?: true` flag rather than a new kind. This keeps the op union minimal and the applyOpInMemory dispatch simple:

```ts
| { kind: 'reject'; path: string; reason: string; held?: true }
```

Change the escalate push to:

#### 3e. Modify `applyOpInMemory`

Add a `pin` case before the final reject/retract:
```ts
if (op.kind === 'pin') {
  const art = arts.get(op.path);
  if (!art) return;
  // Re-point the fingerprint to the current input versions.
  const req = requiredInputs(def, arts, art);
  arts.set(op.path, {
    ...art,
    fingerprint: computeFingerprint(arts, req),
    reasons: [
      ...art.reasons,
      {
        at: Date.now(),
        action: 'reject', // closest existing ReasonAction; 'pinned' could be added if desired
        kind: 'structural',
        by: 'engine',
        text: op.reason,
        fromVersion: art.version,
      },
    ],
  });
  return;
}
```

IMPORTANT: the `pin` op does NOT change `acceptance` (stays `'green'`), does NOT bump `version`, does NOT reset stall counters. It only rewrites `fingerprint` and appends an audit entry.

NOTE on `ReasonAction`: the existing set is `'reject' | 'retract' | 'skip' | 'reopen' | 'retry' | 'born-rejected' | 'schema-reject'`. Adding `'pinned'` as a new action value is optional but clean. If added, add it to `ReasonAction` in types.ts. If not, use `'reject'` as the closest action for the audit entry (acceptable since the kind='structural' and text makes it clear). Decision: add `'pinned'` to `ReasonAction` — it is load-bearing for audit clarity and matches the design's language. Similarly add `'held'` or reuse `'reject'` for the escalate reason entry. Use `'reject'` for escalate (the artifact becomes rejected-and-held; the reason text carries the held semantics).

For the `reject` + `held: true` case in `applyOpInMemory`:

#### 3f. Extend `frozen` / `eligibleFirings` to treat held artifacts as non-eligible

The `held` condition: a rejected artifact whose most-recent reason entry has `kind === 'invalidated-irreversible'`. The producer loop must NOT become eligible.

Add a helper:
```ts
/** True if the artifact is held (non-idempotent escalate) — rejected but not auto-re-arming. */
export function isHeld(a: ArtifactData | undefined): boolean {
  if (!a || a.acceptance !== 'rejected') return false;
  return a.reasons.length > 0 && a.reasons[a.reasons.length - 1]!.kind === 'invalidated-irreversible';
}
```

Extend `frozen`:
```ts
function frozen(a: ArtifactData | undefined, loop: LoopDef): boolean {
  return isStalled(a, loop.maxAttempts) || isSchemaStalled(a, loop.maxSchemaFailures) || isHeld(a);
}
```

This is the key gate: by returning `true` from `frozen`, `eligibleFirings` will NOT include any firing that would rebuild a held artifact (the `!frozen(a, loop)` guard in `plainOutputs.filter`, the map path, and the reduce path all gate on it). No other change to `eligibleFirings` is needed.

## Edge cases

1. **Dead input for non-idempotent loop**: `cascadeFromDeadInput` is called when an input is retracted or skipped — these are structural truths, NOT the "inputs moved" path. The effect routing is only on the `moved-version` reject path. Dead-input cascades for non-idempotent loops must still retract/skip as normal. Add a comment to this effect in the code and test it (test g).

2. **Both `terminal: true` and `effect:` set**: `validateDef` must return a hard error (test f). `buildLoop` may silently set both; the error surfaces at validate time.

3. **`onInvalidate` set with `idempotent: true` (or idempotent omitted, defaulting true)**: Accept silently — do NOT route differently. Only `idempotent: false` activates the routing; when idempotent=true the `onInvalidate` value is simply ignored. Document this in a comment.

4. **`onInvalidate: 'frobnicate'` or any other non-pin/escalate string**: Hard error in `validateDef`, message must say named-handler routing is not yet supported and is a planned follow-up (test f).

5. **Collection members and map children**: The held condition applies to any artifact path, including map children. `frozen` is called per-artifact in `eligibleFirings`, so held map children suppress their individual firing correctly without special casing.

6. **Re-arm after held**: `retry` resets `judgmentRejects` and `schemaRejects` and re-owes the artifact. For held artifacts, `retry` must also clear the held condition. The held condition is detected via the last `reasons` entry's `kind`. A `retry` call appends a `retry` reason entry — so after retry, the last reason is no longer `invalidated-irreversible`, and `isHeld` returns false. This works automatically without additional changes, PROVIDED the engine's existing retry implementation appends a reason entry (verify this is true in src/engine.ts before the builder implements). If retry does NOT append a reason, the builder must add a reason-append step or implement a different clearing mechanism. (Note for builder: check `engine.ts` retry path.)

7. **The `pin` ReasonAction**: if added to `ReasonAction`, update all exhaustiveness checks in the codebase (grep for ReasonAction usage).

## Docs

### docs/design.md

Add a new section after the current `§16 Generated outputs` (which is after §15):

```
## §17 The effect contract (effect:)

- **§17.1** — a loop may declare `effect: { idempotent?, onInvalidate? }` to control
  how the forward cascade routes when the loop's green artifact's inputs move.
- **§17.2 idempotent (default true)** — when true, re-deriving the artifact is safe;
  the engine re-arms it (structural reject) exactly as it does for any non-terminal
  green today. When false, re-running the loop would cause an unretractable side effect.
- **§17.3 onInvalidate (consulted only when idempotent: false)** — defaults to
  `'escalate'`. Two values:
  - `'pin'`: the artifact stays green; its fingerprint is re-pointed to current input
    versions (the *pinned* condition: §6.3). The producer does not re-fire.
  - `'escalate'`: the artifact becomes rejected-and-held (the *held* condition: §6.3).
    The producer does not auto-re-fire; the debt surfaces as stalled, requiring human
    intervention (retry / accept-as-is / fix upstream).
- **§17.4 terminal: vs effect:** — `terminal: true` is the legacy spelling for
  `effect: { idempotent: false, onInvalidate: pin }` plus the dead-end lint exemption.
  The two coexist; migration of `terminal:` to `effect:` is deferred (design doc §6.5).
- **§17.5 dead-input cascade is not gated by effect:** — when a non-idempotent
  artifact's input goes settled-dead (retracted/skipped), the structural cascade
  (retract/skip) applies regardless of `effect:`. Only the moved-version re-arm path
  routes on effect.
- **§17.6 named-handler routing** — `onInvalidate: <loopName>` is a planned follow-up;
  any non-pin/escalate string is currently a hard validateDef error.
```
