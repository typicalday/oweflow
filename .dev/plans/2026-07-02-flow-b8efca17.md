# Plan: Suffixed reduce consume patterns (`consumes: ["src[*].child"]`)

## Goal

Implement the reserved-but-unimplemented suffixed-reduce grammar. Today a
reduce consume must be a bare glob (`src[*]`): it fires when the seal is
green AND every surviving bare member (`src[i]`) is green. This change adds
an optional single-level suffix (`src[*].child`): when present, the gate
moves one level deeper — the reduce fires when the seal is green AND every
surviving member's CHILD artifact (`src[i].child`, typically a map step's
per-element output) is green, not the bare member itself. This makes
"fire once every element's `.plan` has cleared review" expressible, closing
the gap `examples/workflows/research.yaml` currently documents as a caveat
(synth gates on members, not check verdicts).

Bare `src[*]` behavior is unchanged — 100% backward compatible, this is a
strict grammar extension.

## Design confirmed by reading the source (not guessed)

The reduce arm already mirrors a pattern map mode established for exactly
this kind of suffix-chaining: `mapInputPath(mc, index) = elementPath(mc.stem,
index, mc.suffix)` (src/model.ts:125-127), used at src/model.ts:447 so a map
step can chain off *another* map's per-element output. Suffixed reduce needs
the identical trick one level up: a `reduceInputPath` helper computing
`elementPath(rc.stem, index, rc.suffix)`, used everywhere the reduce arm
currently uses the bare member path.

Three of the touch points listed in the proposal turn out to be automatic
once the core fix lands, because the surrounding machinery is already
generic:

- **`buildOrder` (src/engine.ts:633-679)** builds the `consumes` payload
  generically off `Firing.inputs` (line 641-645: `for (const p of
  f.inputs) { const a = arts.get(p); if (a?.value !== undefined)
  consumes[p] = a.value; }`). If `eligibleFirings`'s reduce arm puts child
  paths into `f.inputs` instead of bare member paths, the order payload is
  correct with no changes to engine.ts.
- **Cascade knock-back (`maintainDecisions`, src/model.ts:556-644)** is
  driven purely by `requiredInputs(def, arts, art)` (line 581) and
  `fingerprintMatches`. It has no reduce-specific code path of its own. Once
  `requiredInputs`'s reduce branch computes child paths instead of bare
  member paths when a suffix is present, a rejected/re-greened child
  automatically knocks the reduce output back — no new cascade code.
- **Model checker (`owenloop check`)** reuses `eligibleFirings` directly
  (confirmed: no separate reduce-firing logic exists outside model.ts) — so
  fixing `eligibleFirings` covers the checker too. Only a new checker test
  is needed, not new checker logic.

So the real change is smaller than "touch 6 files' worth of logic" — it's
one parser fix, one small helper + 3 call-site edits in model.ts, one bug
fix in defs.ts's prefix rewriter, plus rendering/lint/docs/tests.

## Correction to the proposal's premise

The proposal says "Existing rule 'at most one reduce-style consume per step'
(`reduceConsume` uses `.find`) stays as-is." That's slightly imprecise:
`reduceConsume` (src/model.ts:87-89) is just a first-match accessor and does
NOT itself enforce the one-reduce rule. The actual enforcement is in
`src/defs.ts`'s `validateDef`, ~lines 1002-1009:
```ts
const maps = l.consumes.filter((c) => c.mode === 'map');
const reduces = l.consumes.filter((c) => c.mode === 'reduce');
if (maps.length > 1) errors.push(`step '${l.name}' has more than one map consume`);
if (reduces.length > 1) errors.push(`step '${l.name}' has more than one reduce consume`);
if (maps.length && reduces.length) {
  errors.push(`step '${l.name}' mixes a map and a reduce consume (pick one shape)`);
}
```
This is stem/suffix-agnostic (counts by `mode`, not `stem`+`suffix`), so
`src[*]` + `src[*].child` on the same step is already correctly rejected as
"more than one reduce consume" today, with no change needed — just don't
assume `.find` is the enforcement point when touching this area.

## Files to touch

### 1. `src/paths.ts` — parser

Current (lines 58-76):
```ts
export function parseConsume(raw: string): ConsumePattern {
  const r = raw.trim();
  let m = MAP_RE.exec(r);
  if (m) {
    return { raw: r, mode: 'map', stem: m[1] as string, binder: m[2] as string, suffix: m[3] as string };
  }
  m = REDUCE_RE.exec(r);
  if (m) {
    if ((m[2] as string) !== '') {
      throw new Error(`reduce pattern must glob the whole set, no suffix: '${raw}'`);
    }
    return { raw: r, mode: 'reduce' as ConsumeMode, stem: m[1] as string, suffix: '' };
  }
  ...
}
```

`REDUCE_RE = /^(.*?)\[\*\](.*)$/` (line 55) already captures the suffix into
`m[2]`. Change: instead of throwing on non-empty suffix, validate it is
**empty or single-level** (`.identifier`, matching the MAP produce suffix
shape used elsewhere — e.g. `.formatcheck`) and accept it; reject
multi-level (`.a.b`) or malformed suffixes (must start with `.`, followed by
one identifier segment, no further dots or brackets) with a `DefError`-style
message in the existing terse style.

Suggested shape:
```ts
const SUFFIX_RE = /^(?:|\.[A-Za-z_][\w-]*)$/; // "" or ".identifier" — one level only
...
m = REDUCE_RE.exec(r);
if (m) {
  const suffix = m[2] as string;
  if (!SUFFIX_RE.test(suffix)) {
    throw new Error(`reduce suffix must be empty or a single '.child' level: '${raw}'`);
  }
  return { raw: r, mode: 'reduce' as ConsumeMode, stem: m[1] as string, suffix };
}
```
Check the codebase's actual identifier character set for `.child`-style
suffixes before hardcoding `SUFFIX_RE` — grep how MAP_RE/`m[3]` suffixes are
validated elsewhere (they currently are NOT re-validated by regex, they're
just captured raw) and match that convention rather than inventing a
stricter one. If produce-side map suffixes accept arbitrary
`.foo.bar`-free single segments only by convention/test coverage rather
than a regex, keep parity — don't be stricter than the existing map suffix
grammar without reason.

Also update `matchConsume` (lines 107-121): the reduce branch currently
hardcodes `el.suffix === ''` (line 120) to match only bare members — that
function's contract is "does concrete artifact `path` match this consume
pattern", and for a suffixed reduce pattern the matching concrete paths are
the *child* paths (`src[i].child`), not the bare members. Change line
119-120 to compare against `pat.suffix` instead of hardcoding `''`:
```ts
// reduce: matches every member; suffix must match (bare reduce: suffix "")
return el.suffix === pat.suffix ? { index: el.index } : null;
```
This is a one-line change and keeps bare reduce behavior identical (both
sides are `''`). Confirm no other caller relies on the old hardcoded-`''`
behavior (grep `matchConsume` call sites) before changing it — if
`matchConsume` isn't actually used anywhere in the reduce eligibility path
(model.ts's reduce arm uses `members()` + direct path construction, not
`matchConsume`), this is still worth fixing for consistency/correctness but
double check it doesn't silently change unrelated behavior.

### 3. `src/model.ts` — the core semantics

**(a) New helper**, next to `mapInputPath` (line 125-127):
```ts
/**
 * The concrete input artifact a reduce fan-in gates on for member index `i`.
 * Normally the bare collection member `stem[i]`. But when the reduce consume
 * carries a suffix (`stem[*].child` — fan-in over a map step's per-element
 * output), the gate is that suffixed per-element child `stem[i].child`, not
 * the bare member. Mirrors mapInputPath's suffix-chaining trick one level up.
 */
export function reduceInputPath(rc: ConsumePattern, index: number): string {
  return elementPath(rc.stem, index, rc.suffix);
}
```

**(b) `requiredInputs`** (lines 257-263) — the resting-inputs/fingerprint
computation cascade relies on:
```ts
const rc = reduceConsume(step);
if (rc) {
  const live = members(arts, rc.stem).filter((m) => !isSettledOut(m));
  return [...live.map((m) => m.path), sealPath(rc.stem), ...plain];
}
```
Change `live.map((m) => m.path)` to route through `reduceInputPath` when a
suffix is present:
```ts
const rc = reduceConsume(step);
if (rc) {
  const live = members(arts, rc.stem).filter((m) => !isSettledOut(m));
  const memberInputs = live.map((m) => {
    const el = parseElement(m.path);
    return el ? reduceInputPath(rc, el.index) : m.path;
  });
  return [...memberInputs, sealPath(rc.stem), ...plain];
}
```
(`m.path` here is always a bare member per `members()`'s contract, so
`parseElement` always succeeds and the `: m.path` fallback is defensive
only — keep it for symmetry with how map-child handling does the same at
line 251-254, but it should be structurally unreachable.)

**(c) `eligibleFirings` reduce arm** (lines 462-488):
```ts
} else {
  // reduce
  const rc = reduceConsume(step);
  if (rc && plainSatisfied) {
    const seal = arts.get(sealPath(rc.stem));
    if (isGreen(seal)) {
      const mem = members(arts, rc.stem);
      const live = mem.filter((m) => !isSettledOut(m));
      if (!live.some((m) => !isGreen(m))) {
        ...
        firings.push({
          step: step.name,
          key: '',
          inputs: [...live.map((m) => m.path), sealPath(rc.stem), ...plainPaths],
          outputs: outs,
        });
      }
    }
  }
}
```
Two changes:
1. The eligibility check `!live.some((m) => !isGreen(m))` currently checks
   the bare member's greenness. When `rc.suffix` is non-empty, it must
   check the CHILD's greenness instead — `arts.get(reduceInputPath(rc,
   el.index))`, not the member itself. A child that is owed or rejected
   blocks (it's a debt, per the proposal's spec); a member that has no
   corresponding child artifact registered at all also blocks (owed
   semantics — `isGreen(undefined)` is already `false`, so this falls out
   naturally as long as we look up the right path).
2. The firing's `inputs` must list child paths (via `reduceInputPath`), not
   bare member paths, mirroring the `requiredInputs` fix in (b) — this is
   what makes `buildOrder`'s payload and the cascade fingerprint correct
   automatically.

Suggested rewrite:
```ts
} else {
  // reduce
  const rc = reduceConsume(step);
  if (rc && plainSatisfied) {
    const seal = arts.get(sealPath(rc.stem));
    if (isGreen(seal)) {
      const mem = members(arts, rc.stem);
      const live = mem.filter((m) => !isSettledOut(m));
      const liveInputPaths = live.map((m) => {
        const el = parseElement(m.path)!; // members() only returns bare elements
        return reduceInputPath(rc, el.index);
      });
      const allChildrenGreen = liveInputPaths.every((p) => isGreen(arts.get(p)));
      if (allChildrenGreen) {
        const outs = singletonProduces(step)
          .map((p) => p.stem)
          .filter((p) => {
            const a = arts.get(p);
            return isDebt(a) && !frozen(a, step);
          });
        if (outs.length) {
          firings.push({
            step: step.name,
            key: '',
            inputs: [...liveInputPaths, sealPath(rc.stem), ...plainPaths],
            outputs: outs,
          });
        }
      }
    }
  }
}
```
Note bare reduce (`rc.suffix === ''`) still works identically:
`reduceInputPath(rc, i)` = `elementPath(rc.stem, i, '')` = the bare member
path, so `liveInputPaths` degenerates to exactly what `live.map((m) =>
m.path)` produced before. This is the same "empty suffix is a no-op"
property `mapInputPath` already relies on for bare map.

Empty collection (zero live members): `liveInputPaths` is `[]`,
`.every(...)` on an empty array is vacuously `true` — fires immediately,
matching the proposal's stated requirement and bare reduce's existing
behavior.

**(d) `owenloop check` / model checker** — confirmed it calls
`eligibleFirings` directly with no separate reduce logic, so (c) covers it.
Add a new checker test (see Tests section) rather than new checker code.

**(e) Graph rendering** — dot format (line 1331-1333) and mermaid (line
1409-1410) currently hardcode the edge label `"reduce [*]"`. The
`WorkflowGraph` edge type needs the suffix threaded through if not already
present — check `buildGraph`'s reduce-edge construction (search for where

carry `edge.binder`) to see whether `edge.suffix` (or similar) is already
captured on the edge object; if not, add it there first. Then:
```ts

// mermaid
lines.push(`  ${from} -->|"reduce [*]${mmdEscape(edge.suffix ?? '')}"| ${to}`);
```
Only emit the suffix when non-empty if that reads cleaner, but keeping it
simple (empty string renders as nothing) is fine and matches how map's
`edge.binder ?? '$i'` fallback already works.

### 4. Lint — dangling suffixed-reduce wiring

`src/defs.ts`'s existing "consumed stem must have a producer" check (lines
1057-1071) already validates that a reduce's `stem` is produced as a
collection (`collectionStems.has(c.stem)`), which still applies unchanged to
suffixed reduce (the stem-level check doesn't care about suffix). Add a
**new**, narrower warning next to it (same style/section) for the specific
new failure mode: a suffixed reduce (`src[*].child`) whose `.child` pattern
is not actually produced by any map step over the same stem `src` — i.e. no
step has a `produces:` map pattern with `stem === src` and `suffix ===
'.child'` (compare against `mapProduce(step)` results across all steps, or

This is analogous to the existing dangling-consume warnings' style — a
non-fatal `warn` (lint separates errors from warnings; confirm which bucket
by reading how `deadEndWarnings`/similar non-fatal checks are surfaced vs
the hard `errors.push(...)` calls above) since the wiring might be
intentionally satisfied by a differently-shaped producer, or simply because
lint should warn rather than hard-fail on this class of issue — match
whatever severity convention nearby dangling-consume checks already use.
