# PR5b Build Plan — Mode 2 `calls:` Runtime Cascade-Up

## Goal

Add the engine-internal machinery that makes `calls:` loops actually run: spawn a child workflow instance when the gate is satisfied, machine-green the parent's artifact when the child's declared outcome greens, re-provide child inputs when parent sources move, and re-attach instead of re-spawning on crash/re-tick. All behavior lives in `src/engine.ts`. `src/defs.ts` gains one new cross-def validation. `examples/workflows/delivery.yaml` gains `outputs: [merge]`. No changes to `model.ts`, `store.ts` (beyond what PR5a already added), or any conformance scenario.

## The `maintainCalls` Algorithm (Step-by-Step)

Signature: `private maintainCalls(parentWf: string, def: WorkflowDef, now?: number): void`

This method is called OUTSIDE any open transaction. It opens its own `store.tx()` per mutating action.

```
for each loop L in def.loops where L.calls is set:
  let callsStem = L.produces[0].stem   // the single produced artifact name
  let callsPath = callsStem             // == the parentPath stored in producedBy

  STEP 1 — Gather gate stems and check gate readiness.
    gateStems = Object.values(L.callsInputs ?? {})  // parent artifact names wired as child inputs
    If gateStems is empty: the calls: loop has no input wiring; treat gate as satisfied.
    parentArts = this.artMap(parentWf)
    gateReady = gateStems.every(stem => isGreen(parentArts.get(stem)))
    If NOT gateReady: continue to next loop (child not spawnable yet, parent artifact stays owed).

  STEP 2 — Look up any existing child via reverse index.
    existingChild = this.store.findChildByParent(parentWf, callsPath)

  STEP 3 — SPAWN or RE-ATTACH.
    If existingChild === undefined:
      // SPAWN (M2B-SPAWN): no child exists yet, gate is ready.
      Build seedProvide: Record<string,unknown> = {}
      For each [childInputName, parentArtifactName] in L.callsInputs:
        parentArt = parentArts.get(parentArtifactName)
        if parentArt?.value !== undefined: seedProvide[childInputName] = parentArt.value
      childId = engine.createInstance(L.calls!, {
        producedBy: { parentWf, parentPath: callsPath },
        provide: seedProvide,
      })
      // createInstance opens its own tx internally — this is fine, we are outside any tx.
      existingChild = this.store.getWorkflow(childId)  // re-read to populate existingChild
    // else: RE-ATTACH — existingChild is the already-spawned child; no new spawn.

  STEP 4 — Read child's declared outcome artifact.
    childDef = this.resolveDef(existingChild.def)
    childOutcomeStem = childDef.outputs![0]!   // validated to exist by Phase-2 check
    childArts = this.artMap(existingChild.id)
    childOutcomeArt = childArts.get(childOutcomeStem)

  STEP 5 — RE-PROVIDE if parent gate source moved (M2B-REPROVIDE).
    For each [childInputName, parentArtifactName] in L.callsInputs:
      parentArtNow = parentArts.get(parentArtifactName)
      childInputArt = childArts.get(childInputName)
      // Compare values via deep-equal. If the parent's value changed vs what child holds:
      If parentArtNow?.value !== undefined AND !deepEqual(parentArtNow.value, childInputArt?.value):
        this.provideInput(existingChild.id, childInputName, parentArtNow.value)
        // provideInput opens its own tx and calls settle on the child.

  STEP 6 — MACHINE-GREEN or STAY OWED (M2B-CASCADEUP).
    Re-read childArts (re-provide may have updated them).
    childOutcomeArt = childArts.get(childOutcomeStem)  // re-read
    parentCallsArt = parentArts.get(callsStem)

    If isGreen(childOutcomeArt):
      shouldGreen = false
      If !isGreen(parentCallsArt):
        shouldGreen = true   // parent calls: artifact is still owed
      Else if !deepEqual(childOutcomeArt.value, parentCallsArt.value):
        shouldGreen = true   // child re-greened its outcome with a new value

      If shouldGreen:
        // MACHINE-GREEN (no worker run, no CAS):
        this.store.tx(() => {
          parentArtsInTx = this.artMap(parentWf)
          parentCallsArtInTx = parentArtsInTx.get(callsStem)
          If !parentCallsArtInTx: return  // artifact not materialized yet
          parentDef = def
          fp = computeFingerprint(parentArtsInTx, gateStems)  // gate dimension only
          next: ArtifactData = {
            ...parentCallsArtInTx,
            acceptance: 'green',
            version: parentCallsArtInTx.version + 1,
            value: childOutcomeArt!.value!,
            fingerprint: fp,
          }
          this.store.putArtifact({ ...next, workflow: parentWf })
          this.settle(parentWf, parentDef, now)
        })
        this.fire({ type:'commit', workflow:parentWf, path:callsStem, action:'provide' })
        this.fireSettled(parentWf)
    // else: child outcome not green yet — parent stays owed, nothing to do.
```

**Deep-equal helper**: Use `JSON.stringify` for simple deep comparison of plain JSON objects (no undefined/function values expected in artifact values). Define `function deepEqual(a: unknown, b: unknown): boolean { return JSON.stringify(a) === JSON.stringify(b) }` as a module-level helper in engine.ts.

**Why fingerprint over gateStems only** (M2B-CASCADEUP decision): The gate stems are the parent artifacts wired into the child via `callsInputs`. Recording their versions in the fingerprint means that if a gate input later moves to a new version, the existing cascade's `maintainDecisions` will re-arm the parent's calls: artifact to `owed` (the normal cascade re-arm). This triggers `maintainCalls` to re-provide the moved input to the child. The child's own outcome version is NOT included in the fingerprint — including it would break `fingerprintMatches`' key-count check (the model counts keys and expects them to match the gate stems). The child-outcome re-green trigger is handled by `maintainCalls` itself (the `deepEqual` value comparison in Step 6), not by the cascade.

## Tick Integration (M2B-ARCH)

In `Engine.tick` (currently line 275), add `maintainCalls` invocation at the TOP, BEFORE the existing `store.tx()` block:

```typescript
tick(workflow: string, opts: { now?: number } = {}): TickResult {
  const def = this.defFor(workflow);
  const now = opts.now ?? nowMs();
  // M2B: maintain calls: child instances before the normal tick/reap/claim cycle.
  this.maintainCalls(workflow, def, now);
  return this.store.tx(() => {
    this.settle(workflow, def, now);
    // ... rest unchanged
  });
}
```

The `maintainCalls` call is OUTSIDE the `store.tx()` because `createInstance` opens its own tx internally. Running nested `store.tx()` would deadlock in better-sqlite3 (it does not support nested BEGIN IMMEDIATE). `maintainCalls` itself uses `store.tx()` only for the machine-green step, also outside any outer tx.

## Cascade-Up Prompt Hook (M2B-CASCADEUP)

The cascade-up prompt makes the parent re-evaluate PROMPTLY when the child makes progress (instead of waiting for the next scheduled tick). Hook it in `green` and `close` at the engine operation layer — after the child's commit fires but after its tx has committed.

**In `green`** (after `this.fireSettled(workflow)` at the end of the method, approximately line 509):
```typescript
// M2B cascade-up prompt: if this workflow has a producedBy link, trigger parent maintainCalls.
this.triggerParentIfChild(workflow);
```

**In `close`** (after `this.fire({ type:'closed', ... })` at line 743):
```typescript
// M2B cascade-up prompt: closing a run may advance the child's artifact state.
this.triggerParentIfChild(workflow);
```

**The `triggerParentIfChild` helper**:
```typescript
private triggerParentIfChild(workflow: string): void {
  const wfRow = this.store.getWorkflow(workflow);
  if (!wfRow?.producedBy) return;
  const { parentWf } = wfRow.producedBy;
  // Recursion guard: if we are already in maintainCalls for this parent, skip.
  if (this._inMaintainCalls.has(parentWf)) return;
  const parentDef = this.resolveDef(this.store.getWorkflow(parentWf)?.def ?? '');
  this.maintainCalls(parentWf, parentDef);
  this.fireSettled(parentWf);
}
```

**Recursion guard**: Add a private field `private readonly _inMaintainCalls = new Set<string>()` to Engine. Wrap the body of `maintainCalls` with:
```typescript
if (this._inMaintainCalls.has(parentWf)) return;
this._inMaintainCalls.add(parentWf);
try { /* ...algorithm... */ } finally { this._inMaintainCalls.delete(parentWf); }
```
This prevents `maintainCalls(parent)` → `provideInput(child)` → `triggerParentIfChild(child)` → `maintainCalls(parent)` infinite recursion. Because `provideInput` fires `fireSettled` which calls `triggerParentIfChild` which calls `maintainCalls` on the parent — but the guard catches it.

## Machine-Green Helper Pattern

The machine-green action mirrors `provideInput` but has a different fingerprint. Here is the exact shape:

```typescript
// Inside maintainCalls, Step 6:
const parentCallsArt = this.store.getArtifact(parentWf, callsStem);
if (!parentCallsArt) return;  // not yet materialized by pendingOwed — skip
const gateArts = this.artMap(parentWf);
const fp = computeFingerprint(gateArts, gateStems);
const next: ArtifactData = {
  ...parentCallsArt,
  acceptance: 'green',
  version: parentCallsArt.version + 1,
  value: childOutcomeArt.value!,
  fingerprint: fp,
};
this.store.putArtifact({ ...next, workflow: parentWf });
const parentDef = this.resolveDef(parentWf_def);
this.settle(parentWf, parentDef, now);
```

NOTE: `gateStems` may be empty if `callsInputs` is empty. In that case `computeFingerprint(gateArts, [])` returns `{}` — an empty fingerprint — which is safe. The cascade will not re-arm the artifact (empty fingerprint has no versions to compare), which is correct: there are no inputs to move.

Do NOT set `terminal: true` on the machine-greened artifact — the calls: artifact must be re-armable if gate inputs move.

## Transaction Composition Care

1. `maintainCalls` is called OUTSIDE any open `store.tx()`. It is synchronous. Each spawning action uses `createInstance` (which has its own `store.tx()`) or `provideInput` (which has its own `store.tx()`), or the machine-green block (which opens its own `store.tx()`). None of these nest.
2. `tick` calls `maintainCalls` BEFORE `store.tx(...)` — not inside it. The subsequent `store.tx()` in tick then calls `settle` which reads the updated state.
3. `triggerParentIfChild` is called AFTER a tx has committed (after `fire`/`fireSettled`, which run post-commit). So it is also outside any open tx.
4. `provideInput` (re-provide step) opens its own tx. If called from within the `maintainCalls` loop, it is fine because `maintainCalls` is not in a tx.
5. Never open a `store.tx()` inside an already-open `store.tx()` — better-sqlite3 does not support nested IMMEDIATE transactions.

## Edge Cases

1. **No callsInputs (empty wiring)**: If `loop.callsInputs` is empty, gateStems is `[]`, gate is always satisfied. `deepEqual` comparisons in re-provide step are skipped. Machine-green fingerprint is `{}`. Spawn fires as soon as `maintainCalls` is first invoked.
2. **Calls artifact not yet materialized**: `pendingOwed` in `settle` creates the calls: artifact as `owed`. If `maintainCalls` runs before the first `settle`, `getArtifact(parentWf, callsStem)` may return undefined. Skip the machine-green step silently — the next tick's settle will materialize it.
3. **Child def has no value on its outcome artifact yet**: `childOutcomeArt?.value` is `undefined`. In Step 6, skip the machine-green — only green when the child has an actual value to reflect.
4. **Parent workflow row not found**: `triggerParentIfChild` calls `this.store.getWorkflow(parentWf)` — if it returns undefined (e.g., parent deleted), swallow the error or guard with `if (!parentWfRow)` return.
5. **Reaper interaction**: When the reaper frees a calls: task (if one exists) — note: `calls:` loops emit NO worker orders (eligibleFirings skips them), so they have no task rows and the reaper never touches them. The re-attach guard is structural: `findChildByParent` returning non-null prevents any second spawn.
6. **Child outcome greens then re-arms (test f)**: If a gate input moves, the existing cascade's `maintainDecisions` re-arms the parent calls: artifact to `owed` (fingerprint mismatch). Next `maintainCalls` call re-provides the moved input to the existing child (Step 5). The child re-runs; when its outcome re-greens, `maintainCalls` machine-greens the parent again.
7. **Multiple calls: loops in one parent**: The `for` loop in `maintainCalls` iterates all `loop.calls` loops. Each is independent. Each has its own `findChildByParent` lookup keyed by `callsPath` (the `produces[0].stem`).

### Tests:

**(a) Happy path end-to-end**
```

provide proposal -> tick -> get provision order -> green sandbox -> close
tick -> assert no worker orders for deliver (calls: excluded from firings)
```

**(b) Re-attach — no duplicate child**

**(c) Re-provide on parent input move**
```

```

**(e) Child outcome re-green**
```

```
