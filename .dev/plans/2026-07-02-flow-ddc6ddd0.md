# Plan: harden judge authority scoping in `assertAuthority` (reject path)

## Goal

A judge step declared `judges: <stem>` must be able to `reject()` /
judgment-invalidate ONLY the exact stem named in its own `judges:` marker ‚Äî
never a stem it merely *consumes for context* via `inputs: true`. Today
`assertAuthority` grants reject authority to any step that consumes the
target stem, and a judge synthesized with `inputs: true` consumes the
producer's own input stems in addition to the judged stem. That means a judge
assigned to judge `report` (with `inputs: true`) can currently call
`reject(wf, 'question', judgeStepName, ...)` and it succeeds ‚Äî landing a
judgment-reject (and a `judgmentRejects` bump) on the wrong, unrelated
artifact. This is a real, non-blocking gap flagged by the reviewer of PR #42;
this instance closes it.

Fix scope: one authority check in `src/engine.ts`, one new test in
`test/judges.test.ts`. No CLI/schema/API surface changes. No changes to
`examples/workflows/`. No changes to `docs/proposals/artifact-judge.md`
(untracked, out of tree ‚Äî it will not exist in the builder's worktree; do not
recreate it). No `docs/design.md` edits (see Docs section below).

## Root cause, precisely

- `src/defs.ts`, `synthesizeJudgeSteps` (~line 783-810): for a judge declared
  `inputs: true`, `consumeStems = [pat.stem, ...producerConsumeStems]`
  (line 790) ‚Äî the judge step's synthesized `consumes` includes BOTH the
  judged stem AND the producer's own input stems (e.g. `question`). This is
  intentional (design ¬ß24 / the produce-level `judges:` entry's `inputs?:
  boolean` field, `src/types.ts` ~line 197: "default false: judge sees only
  the judged value") ‚Äî giving the judge "what was asked for" as read-only
  context. The bug is not here; do not change this file.
- `src/engine.ts`, `assertAuthority` (~line 1614-1627): grants authority to
  invalidate `path` to any step where
  `step.consumes.some((c) => c.stem === stem || c.stem === path)` is true ‚Äî
  stem-membership only, no distinction between "this step's judged stem" and
  "this step's context-input stems." For a judge step, both the judged stem
  and (if `inputs: true`) the context-input stems pass this check identically.
- `Engine.reject()` (~line 962): calls `this.assertAuthority(def, by, path,
  'reject')` at line 964 ‚Äî this is the only gate. Only *after* that check
  passes does it separately compute (line 965-966):
  `const judgeStep = def.steps.find((s) => s.name === by); const judgedStem =
  judgeStep?.judges;` ‚Äî and `judgedStem` is used solely for the
  `judgeCasCheck` staleness guard (line 979), never to constrain which `path`
  was an acceptable target in the first place. Net effect: a judge with
  `inputs: true` calling `reject(wf, 'question', judgeStepName, text)` passes
  `assertAuthority` (the judge consumes `question`), then mutates the
  `question` artifact directly (`this.store.getArtifact(workflow, path)` at
  line 970 uses the caller's `path`, not `judgedStem`) ‚Äî bumping `question`'s
  `judgmentRejects` and clearing its ledger, even though the judge was never
  supposed to have a say over `question` at all.

## The fix ‚Äî src/engine.ts

Tighten `assertAuthority` (~line 1614-1627) so that when `by` resolves to a
judge step (`StepDef.judges !== undefined`), the rule is no longer "consumes
the stem" but "the target stem equals this step's own declared `judges:`
stem." Non-judge consumers are completely unaffected ‚Äî their existing
"consumes the stem" rule stays exactly as-is.

```ts
private assertAuthority(def: WorkflowDef, by: Author, path: string, _action: string): void {
  if (by === 'human' || by === 'engine') return;
  const el = parseElement(path);
  const stem = el ? el.stem : path.replace(/\.sealed$/, '');
  const step = def.steps.find((l) => l.name === by);
  if (!step) throw new Error(`unknown actor: ${by}`);

  if (step.judges !== undefined) {
    // ¬ß24: a judge may only invalidate the exact stem named in its own
    // `judges:` marker ‚Äî never a stem it merely consumes for context via
    // `inputs: true` (defs.ts synthesizeJudgeSteps splices the producer's
    // own input stems into a judge's `consumes` for read-only context;
    // that consume edge must grant NO reject authority over those stems).
    if (step.judges !== stem && step.judges !== path) {
      throw new Error(
        `${by} has no authority to invalidate ${path} (it judges \`${step.judges}\`, not \`${stem}\`). ` +
          `A judge may only reject the stem named in its own \`judges:\` marker.`,
      );
    }
    return;
  }

  const consumesIt = step.consumes.some((c) => c.stem === stem || c.stem === path);
  if (!consumesIt) {
    throw new Error(
      `${by} has no authority to invalidate ${path} (it does not consume it). ` +
        `Authority follows the consume edge (¬ß4.1): add \`${stem}\` to ${by}'s \`consumes\` to grant it.`,
    );
  }
}
```

Notes on this shape:

- Mirrors the existing permissiveness of accepting either the parsed `stem`
  or the raw `path` (`step.judges !== stem && step.judges !== path`), matching
  how the pre-existing `consumesIt` check already accepts both forms.
- Keep the throw in the same style/spirit as the existing error (plain
  `Error`, no new error class/type) ‚Äî matches the proposal's ask ("in the
  same shape/spirit as the existing `assertAuthority` throw"). Tests should
  assert on `assert.throws(...)` (optionally with a loose message match like
  `/authority/i`, following the convention already used in
  `test/edge.e2e.test.ts` ‚Äî see "authority: only a consumer (or human) may
  judgment-reject an artifact (¬ß4)" which asserts `assert.match(r.stderr,
  /authority/)` on the CLI's stderr); don't over-couple to exact wording.
- A judge consuming AND judging its own stem (the normal case) still passes:
  `step.judges === stem` is true, function returns normally ‚Äî zero behavior
  change for the common case, including every existing judge test in
  `test/judges.test.ts` (a)-(j2), none of which use `inputs: true`.
- Human (`by === 'human'`) and engine (`by === 'engine'`) authority is
  unchanged ‚Äî the existing early return still short-circuits before reaching
  the judge branch.
- A non-judge step's authority over stems it consumes is unchanged ‚Äî the new
  branch only triggers when `step.judges !== undefined`. Ordinary
  producer/consumer steps (e.g. the merger example in design ¬ß4.1, or the
  `planner`/`pr` case in `test/edge.e2e.test.ts`) fall through to the
  existing `consumesIt` check, byte-for-byte untouched.
- Single choke point: `assertAuthority` is the only caller-facing gate,
  matching the proposal's explicit preference ("prefer putting the tightened
  rule inside `assertAuthority`... so authority logic stays in one place").
  Do not duplicate this check elsewhere (e.g. inside `reject()`'s body).
- `Engine.reject()` itself needs no structural change ‚Äî it already calls
  `assertAuthority` as the very first thing (line 964), before the CAS check,
  before `this.store.tx(...)`, before any artifact read/write. A judge
  targeting the wrong stem now throws immediately and nothing is mutated.
  The pre-existing `judgeStep`/`judgedStem` lookup at lines 965-966 stays
  as-is (still needed for the CAS check at line 979) ‚Äî there is a small,
  acceptable duplication of `def.steps.find((s) => s.name === by)` between
  `assertAuthority` and `reject()`'s own body. Do not refactor to eliminate
  that duplication; keep the diff minimal and localized to `assertAuthority`.

### Approve-path audit (no code change expected)

Confirm ‚Äî per the proposal's explicit ask ‚Äî that `green()`'s judge-approve
branch (src/engine.ts, `if (runStep?.judges)` at ~line 696-728) has no
equivalent gap:

- It derives `judgedStem = runStep.judges` (line 697) directly from the run's
  own claimed step definition (`runStep`, resolved from the run's lease, not
  from user input), and uses `judgedStem` for every subsequent lookup:
  `arts.get(judgedStem)` (line 698), the `judgeCasCheck` call (line 699), and
  the final `store.putArtifact` targets `art` (== `judged`, keyed by
  `judgedStem`) ‚Äî the caller-supplied `path` argument to the public
  `green(workflow, run, path, value)` call is read earlier only in the
  non-judge branch (~line 660, `arts.get(path)`); once the judge branch is
  entered (`runStep?.judges` truthy), `path` is never consulted again.
- Conclusion: a judge's `green()` call cannot land on the wrong stem no
  matter what `path` string the caller passes, because the judge branch
  derives its target from the run's own step definition, not from `path`.
  This is structurally safe already ‚Äî **no code change needed in `green()`.**
  State this conclusion explicitly in the PR description; do not add a
  redundant guard there (the proposal: "add the same guard only if you find
  a real gap" ‚Äî there isn't one).

## The test ‚Äî test/judges.test.ts

Add a new fixture with a judge declared `inputs: true` (so its synthesized
step consumes both the judged stem `report` and the producer's input stem
`question`), plus one new test proving both halves of the fix. Follow the
file's existing conventions closely ‚Äî reuse `makeEngine`/`getArt` (already
defined at lines 49-62), the `import { test } from 'node:test'` /
`assert from 'node:assert/strict'` imports already at the top, and the
`tick().orders.find(o => o.step.includes('.judges.'))` pattern that every
existing test in the file already uses to get the judge run/step rather than
hard-coding the synthesized name string.

### Test

Place after the existing test (j2) at the end of the file:

```ts
// ---- (k) judge authority is scoped to its own judged stem only --------------

test('judges: (k) a judge with inputs:true cannot reject a stem it only consumes for context', () => {
  const { engine, store } = makeEngine([researcherWithInputsJudgeDef()]);
  const wf = engine.createInstance('researcherInputsJudgeDef', { provide: { question: { text: 'why' } } });

  const run1 = engine.tick(wf).orders.find((o) => o.step === 'researcher')!.run;
  engine.green(wf, run1, 'report', { sections: ['a'] });
  engine.close(wf, run1);
  assert.equal(getArt(store, wf, 'report')?.acceptance, 'submitted');

  const tick2 = engine.tick(wf);
  const judgeOrder = tick2.orders.find((o) => o.step.includes('.judges.'))!;
  assert.ok(judgeOrder, 'the inputs:true judge should be eligible on the submitted report');

  const questionBefore = getArt(store, wf, 'question');

  // The judge consumes `question` for context (inputs: true) but does not
  // judge it ‚Äî rejecting it must be refused, not applied.
  assert.throws(() => engine.reject(wf, 'question', judgeOrder.step, 'trying to reject the wrong stem'));

  // `question` is completely untouched by the refused reject.
  assert.deepEqual(getArt(store, wf, 'question'), questionBefore);

  // The judged `report` is untouched too ‚Äî still submitted, no ledger change.
  const reportAfter = getArt(store, wf, 'report');
  assert.equal(reportAfter?.acceptance, 'submitted');
  assert.equal(reportAfter?.judgmentRejects, 0);

  // The SAME judge rejecting its own declared stem (`report`) still works.
  const res = engine.reject(wf, 'report', judgeOrder.step, 'actually reject the judged stem');
  assert.equal(res.outcome, 'rejected');
  engine.close(wf, judgeOrder.run);

  const reportRejected = getArt(store, wf, 'report');
  assert.equal(reportRejected?.acceptance, 'rejected');
  assert.equal(reportRejected?.judgmentRejects, 1);
});
```

The two required proof points (per the proposal):

1. The judge calling `reject` on the producer-input stem it consumes via
   `inputs: true` but does not judge is refused (throws); the wrongly-targeted
   artifact (`question`) is provably untouched (`deepEqual` against its
   pre-call snapshot), and the judged `report` stays `submitted` with
   `judgmentRejects` still `0`.
2. The same judge rejecting its own declared stem (`report`) still works
   normally: `outcome: 'rejected'`, `judgmentRejects` bumps to `1`,
   `acceptance` becomes `'rejected'`.

Minor implementation freedom for the builder: exact assertion style
(`deepEqual` vs. field-by-field equals) and whether `assert.throws` takes a
message-matching second argument are both fine either way ‚Äî match whatever
reads most consistently with the rest of the file. Don't hard-code the
judge's synthesized step name (`researcher.report.judges.completeness`)
anywhere; always derive it from `tick().orders`.

## Edge cases / non-goals to keep in mind while implementing

- **Judge with `inputs: false` (default)**: consumes only the judged stem, so
  `step.judges === stem` is always true for the one stem it can target ‚Äî
  behavior identical before/after. Existing tests (a)-(j2) all use plain
  `researcherDef` (no `inputs: true`) and must keep passing unchanged as the
  regression guard for this case.
- **Non-judge consumer rejecting a stem it legitimately consumes** (e.g. the
  `planner`/`pr` authority test in `test/edge.e2e.test.ts`, or the merger/`pr`
  example in design ¬ß4.1): completely unaffected ‚Äî the new branch is gated on
  `step.judges !== undefined`, so ordinary steps never enter it.
- **`by === 'human'` / `by === 'engine'`**: unchanged, short-circuits before
  the judge branch is ever reached.
- Do not touch `synthesizeJudgeSteps` in `src/defs.ts` ‚Äî its `consumes` list
  is correct as designed; the fix belongs entirely in the authority check,
  not in what a judge consumes.
- Do not touch `judgeCasCheck` or any CAS logic in `reject()`/`green()` ‚Äî
  that's a separate, already-fixed concern (PR #42, staleness/races). This
  change is about *which stem* a judge may target, not *when* or *staleness*.
- Two sibling judges on the same produce (one `inputs: true`, one not, as in
  `researcherDef`'s `completeness`/`rigor` pair) are unaffected by each
  other ‚Äî the check is purely a function of the acting step (`by`) itself.
